Chopstick Problem

Keypoints:

Petri Net Representation
Ei: Eating state of a philosopher.
Mi: Meditating state of a philosopher.
Ci: Chopsticks availability.
Tokens: Represent active states or availability of chopsticks.
Deadlock Risk: If all philosophers prioritize the left chopstick, no one can acquire both chopsticks, leading to a deadlock.

Strategy 1: Hungry State. After meditating, philosophers enter a "hungry" state. They first take the left chopstick, and if the right is available, they eat. Problem: Deadlock can still occur if no right chopsticks are available.

Strategy 2: Gentleman Strategy One philosopher (the "gentleman") does not prioritize the left chopstick. This reduces the chance of deadlocks by allowing others to eat first.
These strategies aim to manage resources (chopsticks) effectively and reduce deadlock risks.


Current Status:

  • Who eats: E2 and E4
  • Who meditates: M1, M3, M5
  • Available chopstick: Only C5

What happens when E2 and E4 finish eating?

  • E2 and E4 become M2 and M4 (meditating).
  • M1, M3, and M5 get chopsticks and start eating (become E1, E3, and E5).

Is it possible to avoid deadlock?

Yes, by implementing strategies like the "gentleman" (Strategy 2) strategy or assigning priorities to avoid cyclic dependencies, the system can avoid deadlocks and ensure all philosophers get a chance to eat.

Good Architecture

Keywords:
• Process recommendations
• Structural recommendations

Process Recommendations:
Conceptual integrity and technical consistency: Good communication between architects and the development team to avoid impractical designs ("ivory tower").
Architecture should prioritize clearly defined quality attribute requirements.
Architecture should be documented using views that align with stakeholder needs.
The architecture should be evaluated for its effectiveness in meeting critical quality attributes.
Incremental implementation: Build the system step-by-step to stay flexible.

Structural Recommendations:
Well-defined modules: Divide the system into parts with clear responsibilities.
Information hiding and separation of concerns: Ensure each module only knows what it needs to perform its task.
Software quality attributes should be achieved using well-known architectural patterns and tactics.
Independence: The architecture should never depend on a particular version of a commercial product or tool.
Data Separation: Keep data-producing and data-consuming modules separate.
Simplicity: Use simple designs to enhance understanding and reliability.
Flexibility: Allow easy reassignment of processes.
Resource Control: Minimize areas where resources might conflict.

Key Distinction:
Process focuses on how the architecture is created and managed (teamwork, planning, evaluation).
Structure focuses on the technical design of the system (modules, independence, simplicity).

Tradeoffs of Microservice Architecture

Microservice architecture divides a system into small, independent services, each addressing specific needs. While flexible, it introduces notable challenges:

  • Increased Overhead: Network-based communication introduces latency, costs, and potential performance issues.
  • Complex Transactions: Synchronizing actions across distributed systems is difficult, especially for consistent operations.
  • Technological Diversity: Multiple technologies increase maintenance complexity and require diverse expertise.
  • Lack of Visibility: Managing numerous microservices complicates system-wide understanding and troubleshooting.
  • Design Complexity: Defining service responsibilities and optimal granularity is challenging and requires expertise.
  • Deployment Planning: Independent deployment demands an architecture designed to support this flexibility.

Analogy: Building a house with no unified plan, where every builder works independently, can result in chaos and miscoordination.

What is MVC?

Model View Controller

... is a software design pattern for developing web applications. It is an example of a usability pattern with various variants, such as:

It separate the presentation layer from bussines logic and data layer

  • MVP (Model-View-Presenter)
  • MVVM (Model-View-ViewModel)
  • MVA (Model-View-Adapter)
  • MVT (Model-View-Template)

Key Components:

  • Model: Manages the data and business logic.
  • View: Represents the visual interface for the user.
  • Controller: Handles user input, updates the model, and refreshes the view.

Benefits:

  • Separates presentation logic from business logic.
  • Enables parallel development of components.
  • Simplifies maintenance and future enhancements.
  • Promotes reusability of models and views.

Tradeoffs:

  • Can be burdensome for complex UIs with many interconnected components.
  • Adds upfront complexity for simple UIs.
  • Introduces slight latency in user interactions.

Challenges:

  • Difficult for beginners to understand and implement.
  • Complex communication between components in large-scale applications.

Test-Driven Development (TDD)

Process:

  1. Think: Start with a test to define the desired behavior.
  2. Red: Write a test using the public interface; it will fail initially.
  3. Green: Write minimal code to make the test pass.
  4. Refactor: Improve the existing code while ensuring tests continue to pass.
  5. Repeat: Add new behavior by starting the cycle again.

Benefits:

  • Quickly identifies mistakes.
  • Improves overall code quality.
  • Reduces bugs and pinpoints them as they appear.

BPNM, tell me all about it.

Business Process Model and Notation

BPMN is a standard for modeling business processes. It uses graphical flowcharts with key elements like Activities, Events, and Gateways to visualize and specify process workflows.


Benefits: BPMN makes business processes easier to understand by using simple flowcharts. It creates a shared language between teams, works for many industries, helps find and fix problems, and supports tools for automation.


BPMN has 4 kinds of diagrams:
• Process diagrams
• Collaboration diagrams
• Conversation diagrams
• Choreography diagrams

BPMN Process diagram

Represents regular flow between tasks, events and decision points to complete a process in the company.

BPMN Collaboration diagram

Represents message flows between processes or entities (e.g., customers or partners). Collaboration diagrams include 2+ pools representing participants. Interactions shown as message exchanges. Pools can be empty or show internal processes.

BPMN Conversation diagram

- Represents groups of messages (communications) and their relationships
- Top-level view of BPMN collaboration diagrams
- Process landscape and high-level interactions
- Conversation nodes are displayed as hexagons

BPMN: Choreography Diagram

- Represents interactions between participants (tasks, users, or resources) and the messages exchanged.
- Each step involves a choreography activity with 2 participants.
- The initiator is shown with a light color, and the recipient is shown with a dark color.
- An envelope icon represents a message sent by one of the parties.
